Scenario 3: Runtime Model Capability Mismatch

This scenario illustrates what happens when a user selects a model that doesn't support the required capability. The system detects this mismatch and can either fall back automatically or ask the user for input.

User Interface
What can you tell me about this chart of housing prices? [IMAGE ATTACHED]
Model Capability Mismatch

GPT-3.5 Turbo doesn't support vision capabilities required for analyzing images.

Please select an alternative model with vision support:

System Implementation
// User has selected 'gpt-3.5-turbo' but sent an image // The system detects this runtime mismatch const userSelectedProvider = 'openai'; const userSelectedModel = 'gpt-3.5-turbo'; const requiredCaps = [ModelCapability.VISION]; // From QueryAnalyzer // Validate model's capability const validationResult = await art.reasoningEngine.validateModelForCapabilities( userSelectedProvider, userSelectedModel, requiredCaps ); if (!validationResult.valid) { // Query ModelRegistry for alternatives const alternatives = await art.modelRegistry.suggestAlternatives( requiredCaps ); // Show UI dialog with alternatives (user story) const selectedAlternative = await showModelSelectionDialog( `${validationResult.requestedModel} doesn't support vision capabilities`, alternatives ); if (selectedAlternative) { // Process with the user's chosen alternative await art.process({ query: "What can you tell me about this chart? [Image]", threadId: 'thread-123', options: { preferredProvider: selectedAlternative.provider, preferredModel: selectedAlternative.modelId, } }); // Optionally save preference if user checked "remember" if (selectedAlternative.remember) { await art.stateManager.updateThreadConfig('thread-123', { reasoning: { modelPreferences: { preferredModels: { [ModelCapability.VISION]: { provider: selectedAlternative.provider, modelId: selectedAlternative.modelId } } } } }); } } }

Scenario 4: Runtime Fallback on Error

This scenario shows how the system handles errors from a provider (like rate limits) and allows users to choose fallback options at runtime rather than experiencing a complete failure.

User Interface
Write a detailed essay about sustainable urban development approaches in emerging economies.
🔍 Model Selected: Anthropic / Claude 3.7 Sonnet (based on REASONING capability)
⚠️ API Error: Anthropic API rate limit exceeded (Status: 429)
Model Error: Choose Fallback Option

The selected model Anthropic / Claude 3.7 Sonnet encountered an error:

Rate limit exceeded. Please try again in a few minutes.

How would you like to proceed?

System Implementation
// Initial call fails with provider error try { const result = await art.process({ query: "Write a detailed essay about sustainable urban development...", threadId: 'urban-research-thread', options: { preferredProvider: 'anthropic', preferredModel: 'claude-3-7-sonnet-20250219' } }); } catch (error) { // Error is caught by the ModelRouter if (error.type === 'PROVIDER_ERROR' && error.providerErrorCode === 429) { // Rate limit error, system decides how to handle // If automatic fallback is disabled, or user has configured // runtime fallback selection: const fallbackOptions = await art.modelRegistry.suggestFallbackModels( 'anthropic', 'claude-3-7-sonnet-20250219', [ModelCapability.REASONING] ); // Show fallback selection dialog const fallbackChoice = await showFallbackDialog( error.message, fallbackOptions, { allowRetry: true, allowCancel: true } ); if (fallbackChoice) { if (fallbackChoice.action === 'USE_FALLBACK') { // Process with the selected fallback await art.process({ query: "Write a detailed essay about sustainable urban development...", threadId: 'urban-research-thread', options: { preferredProvider: fallbackChoice.provider, preferredModel: fallbackChoice.modelId } }); // Optionally save fallback preference if (fallbackChoice.rememberFallback) { await art.stateManager.updateGlobalFallbackPreference( 'anthropic', fallbackChoice.provider, fallbackChoice.modelId ); } } else if (fallbackChoice.action === 'RETRY') { // Wait and retry with same model setTimeout(() => { art.process({ query: "Write a detailed essay about sustainable urban development...", threadId: 'urban-research-thread', options: { preferredProvider: 'anthropic', preferredModel: 'claude-3-7-sonnet-20250219' } }); }, 30000); // 30 second delay } // Else: request was cancelled } } }

Key Component Relationships

Visualizing how ModelRegistry, ModelRouter, ModelSocket, and provider adapters work together to provide a robust, flexible model management system.

ModelRegistry

  • Role: Central catalog of all available models and their capabilities
  • Key Functions: Model validation, capability matching, fallback suggestions
  • Interactions: Used by ModelRouter to find suitable models for capabilities

ModelRouter

  • Role: Intelligent router selecting between providers/models based on requirements
  • Key Functions: Selection logic, fallback handling, error categorization
  • Interactions: Works with ModelRegistry, StateManager (preferences), ProviderFactory (adapters)

ModelSocket

  • Role: Communication channel for model-related events to the UI layer
  • Key Functions: Event notification, filtering (by type, threadId)
  • Interactions: Receives events from ModelRouter and ProviderAdapters

Provider Adapters

  • Role: Provider-specific implementation of API interactions
  • Key Functions: API calls, error normalization, token counting
  • Interactions: Created by ProviderFactory, used by ModelRouter, report to CostTrackingService

Together, these components create a robust system that:

  • ✓ Allows fine-grained user control over model selection at multiple levels
  • ✓ Automatically detects and handles capability mismatches
  • ✓ Provides fallback options when errors occur
  • ✓ Gives transparency into model selection decisions
  • ✓ Normalizes provider-specific errors into a consistent format
